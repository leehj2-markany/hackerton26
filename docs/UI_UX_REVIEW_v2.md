### Issue 1: Slack 채널 답변이 챗봇으로 전달되지 않음
- **재현 경로**: 에스컬레이션 → Slack 채널 생성 → 담당자가 채널에서 답변 → 챗봇에 표시 안 됨
- **근본 원인**: `escalateCase()`가 생성한 `channelId`를 `sendSlackQuestion()`에 전달하지 않음. React state(`slackChannelId`)는 비동기 업데이트라 `setSlackChannelId()` 직후에도 이전 값(null) 참조. 결과적으로 `sendSlackQuestion`이 null channelId로 호출되어 새 채널을 또 생성하거나 기본 채널로 전송 → 폴링 채널과 답변 채널 불일치
- **추가 원인**: 초기 폴링 루프(90초) 타임아웃 후 도착하는 답변을 캡처하는 메커니즘 없음
- **영향도**: Critical — 핵심 기능(담당자 실시간 답변) 불가
- **상태**: ✅ 수정 완료 (v2: 2026-03-12)
- **수정 내용**:
  1. `escalationResult?.data?.channelId`를 즉시 `slackChannelId`에 반영 + `activeChannelId` 로컬 변수로 `sendSlackQuestion`에 전달
  2. 에스컬레이션 모드 백그라운드 폴링 `useEffect` 추가 — 10초 간격으로 지속 폴링, `seenSlackTsRef`로 중복 방지
  3. 초기 폴링 루프에서도 `seenSlackTsRef`에 등록하여 백그라운드 폴링과 중복 방지
- **파일**: `Chatbot.jsx`

### Issue 9: "1개 중 0개 질문에 대한 답변이 완료되었습니다" — 실제 답변이 있는데 0개로 카운트

- **현상**: 이현진(SE)이 🟢 LIVE로 "drm 견적과 자료 보내드릴게요. 몇 유저 사용하실건가요?..."라고 실제 답변했음에도, 채소희 마무리 메시지에서 "1개 중 0개 질문에 대한 답변이 완료되었습니다"로 표시
- **재현 경로**: 에스컬레이션 → 이현진 LIVE 답변 도착 → 채소희 "1개 중 0개" 메시지 표시
- **근본 원인 추정**: `answeredCount` 변수가 초기 폴링 루프(90초) 내에서만 증가하는데, LIVE 답변이 백그라운드 폴링(`useEffect`)에서 감지된 경우 `answeredCount`에 반영되지 않음. 초기 폴링 루프가 타임아웃(90초)으로 종료된 후 백그라운드 폴링이 답변을 캡처하면, 이미 `completionText`는 `answeredCount=0`으로 생성된 상태
- **영향도**: 🔴 Critical — 고객이 "답변이 왔는데 왜 0개라고 하지?"라고 혼란. 서비스 신뢰도 저하
- **수정 방향**: `answeredCount`를 로컬 변수가 아닌 ref 또는 state로 관리하여, 백그라운드 폴링에서 감지된 답변도 카운트에 반영. 또는 `completionText` 생성 시점에 실제 표시된 LIVE 메시지 수를 `messages` state에서 직접 카운트

### Issue 10: 채소희 "송인찬님이 현재 다른 업무 중입니다" — 배정되지 않은 담당자 타임아웃 메시지

- **현상**: 서브질문이 이현진에게만 배정되었는데, 채소희가 "송인찬님이 현재 다른 업무 중입니다. 슬랙 채널에서 확인 후 답변드릴 예정입니다"라고 표시
- **재현 경로**: 에스컬레이션 → 서브질문 1개(이현진 배정) → 이현진 답변 완료 → 채소희가 송인찬 타임아웃 메시지 표시
- **근본 원인 추정**: `realAgentNames`가 서브질문의 `assignee`뿐 아니라 채널에 초대된 모든 에이전트를 포함하거나, 에스컬레이션 시 기본 에이전트 목록(송인찬, 이현진, 박우호)이 모두 `assignees`에 포함되어 서브질문과 무관하게 모든 실제 에이전트의 답변을 기다리는 구조
- **영향도**: 🟡 Medium — 고객 입장에서 "송인찬은 왜 나오지? 내 질문과 관계없는데"라는 혼란
- **수정 방향**: 타임아웃 메시지를 서브질문에 실제 배정된 `realAgentNames`에 한정. 채널 초대는 전체 팀이 되더라도, 답변 대기 및 타임아웃 안내는 해당 서브질문 담당자만 대상으로

### Issue 11: 이현진 LIVE 답변이 채소희 마무리 메시지보다 위에 표시 — 시간순 역전

- **현상**: 스크린샷에서 이현진 LIVE 답변(오후 4:40) → 채소희 "송인찬님 다른 업무 중"(오후 4:41) → 채소희 "1개 중 0개 답변 완료"(오후 4:41) 순서로 표시. 이현진 답변이 있는데도 마무리 메시지가 "0개 완료"로 나오는 것은 시간순은 맞지만 로직이 답변을 인식하지 못한 것
- **근본 원인**: Issue 9와 연결. 초기 폴링 루프에서 이현진 답변을 감지했지만 `answeredCount` 증가가 `answeredRealAgents.add()` 이후 `answeredCount++` 경로를 타지 않았거나, 백그라운드 폴링에서 감지된 경우 초기 루프의 `answeredCount`에 반영 안 됨
- **영향도**: Issue 9와 동일 (중복 이슈, 같은 근본 원인)

### Issue 12: 박우호(가상 에이전트) 제거 — AI 봇이 담당자처럼 답변하는 것은 불필요

- **현상**: 박우호(개발리더)가 가상 에이전트(VIRTUAL_AGENTS)로 LLM이 생성한 답변을 마치 실제 담당자처럼 표시. 스크린샷에서 박우호가 "죄송합니다, 현재 AI 응답 생성에 일시적인 문제가 발생했습니다..."라는 에러 메시지를 담당자 말풍선으로 표시
- **문제**: 고객 입장에서 AI가 이미 답변하고 있는데, 또 다른 AI가 사람인 척 답변하는 것은 혼란스럽고 불필요. 에러 발생 시 담당자 이름으로 에러 메시지가 나오면 신뢰도 저하
- **영향도**: 🟡 Medium — UX 혼란 + 불필요한 복잡성
- **수정 방향**: `VIRTUAL_AGENTS`에서 박우호 제거. 가상 에이전트 LLM 답변 생성 로직 자체를 제거하거나, 실제 Slack 에이전트(송인찬, 이현진)만 답변 대기하도록 변경
- **관련 코드**: `Chatbot.jsx` — `VIRTUAL_AGENTS` 배열, `virtualPromises` 로직

### Issue 13: 채소희 개입 과다 — 채널 연결 시에만 개입하고 이후 메시지 제거

- **현상**: 채소희(고객센터)가 에스컬레이션 전 과정에 걸쳐 메시지를 보냄: 입장 인사 → 질문 분류 안내 → 추가 질문 전달 확인("네, 담당자분들께 바로 전달하겠습니다!") → 타임아웃 안내("송인찬님이 현재 다른 업무 중") → 마무리("1개 중 0개 답변 완료") → 종료 인사. 채팅창이 채소희 메시지로 가득 차서 실제 담당자 답변이 묻힘
- **문제**: 고객은 담당자 답변을 보러 왔는데, 채소희의 안내 메시지가 너무 많아 핵심 정보(담당자 LIVE 답변)를 찾기 어려움. 채팅 UX가 혼란스러움
- **영향도**: 🟡 Medium — 정보 과부하, 핵심 답변 가독성 저하
- **수정 방향**: 채소희 역할을 "채널 연결 + 담당자 초대"까지만 한정. 이후 질문 전달, 타임아웃 안내, 마무리 메시지는 시스템 메시지(회색 이탤릭)로 대체하거나 최소화. 구체적으로:
  - ✅ 유지: 채소희 입장 인사 + "전문가분들을 연결해 드릴게요"
  - ❌ 제거/축소: 질문 분류 상세 안내, 추가 질문 전달 확인, 타임아웃 안내, 마무리 카운트 메시지
  - 마무리 버튼("더 물어볼게요" / "충분합니다")은 시스템 UI로 표시 (채소희 말풍선이 아닌 독립 UI)
- **관련 코드**: `Chatbot.jsx` — `handleEscalation()` 내 채소희 메시지 다수, `handleFollowUpQuestion()` 내 채소희 전달 확인 메시지

### Issue 14: 에스컬레이션 완료 후 입력창 placeholder가 "담당자에게 질문하기..."로 유지

- **현상**: 채소희 마무리 메시지 + "더 물어볼게요" / "충분합니다" 버튼이 표시된 상태에서, 입력창 placeholder가 "담당자에게 질문하기..."로 표시. 이 시점에서 사용자가 텍스트를 입력하면 어디로 가는지 불명확
- **재현 경로**: 에스컬레이션 완료 → 마무리 버튼 표시 → 입력창 "담당자에게 질문하기..."

- **영향도**: 🟡 Medium — UX 혼란. 버튼으로 선택해야 하는 시점에 텍스트 입력이 가능해 보임
- **수정 방향**: `showContinueOrEnd` 상태일 때 placeholder를 "위 버튼을 선택해 주세요"로 변경하거나, 입력을 일시 비활성화

### Issue 15: AgentStatus 패널 "담당자 연결 중" — 이미 답변 완료된 후에도 연결 중 표시 + 담당자 목록 불일치

- **현상**: 이현진(SE)이 이미 LIVE 답변을 완료하고, 채소희 마무리 메시지 + 버튼까지 표시된 상태인데, 하단 AgentStatus 패널이 여전히 "담당자 연결 중 🏗 2/4"로 표시. 패널 내 담당자 목록에 채소희 ✅, 송인찬 ✅만 보이고, 실제 답변한 이현진은 목록에 없음
- **문제점 2가지**:
  1. 에스컬레이션 시퀀스가 끝났는데 "연결 중" 상태가 해제되지 않음 — `showAgentStatus`가 `false`로 전환되지 않거나, 패널 자체가 닫히지 않음
  2. 담당자 목록이 서브질문 배정과 불일치 — 이현진이 배정되어 답변했는데 패널에 없고, 배정되지 않은 송인찬이 ✅로 표시
- **근본 원인 추정**: `agents` state 배열에 담당자를 추가하는 로직이 `assignees`(서브질문 기반)가 아닌 기본 에이전트 목록을 사용하거나, 입장 순서에 따라 일부만 추가됨. 또한 에스컬레이션 완료 후 `setShowAgentStatus(false)` 호출이 누락
- **영향도**: 🟡 Medium — 고객이 "아직 연결 중인가? 이현진은 어디 갔지?"라고 혼란
- **수정 방향**: 
  - 에스컬레이션 시퀀스 완료(Step 8 마무리) 시 `setShowAgentStatus(false)` 호출하여 패널 닫기
  - 또는 패널을 "연결 중"이 아닌 "연결 완료" 상태로 전환하여 참여한 담당자 목록 표시
  - `agents` 배열을 서브질문 `assignees` 기반으로 정확히 구성
- **관련 코드**: `Chatbot.jsx` — `showAgentStatus`, `agents` state, `AgentStatus` 컴포넌트
