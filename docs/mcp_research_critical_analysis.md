# MCP 데이터 리서치 제안 - 비판적 분석

## 📋 제안 내용
"데모에서 가상의 사용자들 질문은 지라, 컨플런스, 슬랙, 구글드라이브, 지메일 MCP로 리서치해서 데이터 증강하자"

---

## ✅ 좋은 점

### 1. 현실성 극대화
- 실제 고객 질문 패턴 사용 → 데모가 가상이 아닌 실제처럼 보임
- 심사위원이 "이건 진짜 쓸 수 있겠다"고 느낄 가능성 높음

### 2. 차별화 포인트
- 다른 팀: 가상 시나리오
- 우리 팀: 실제 데이터 기반
- 발표에서 "Jira 이슈 1,234건 분석" 같은 구체적 숫자 제시 가능

### 3. 답변 품질 향상
- 실제 해결 사례 기반 → AI 답변이 더 정확하고 구체적
- 참조 문서도 실제 문서 → 신뢰도 높음

### 4. 고객 이력 데이터의 진정성
- SK하이닉스 프로필에 실제 과거 질문 포함 → 맥락이 자연스러움
- "2월에 문의하신 성능 이슈" 같은 표현이 설득력 있음

## ⚠️ 문제점 및 리스크

### 1. 시간 투자 대비 효과 (가장 큰 문제)
**문제**: 
- 리서치 2.5시간 + 데이터 정리 1시간 = 총 3.5시간
- 현재 53시간 예산에서 3.5시간은 6.6%
- 이 시간을 핵심 기능 구현에 쓰는 게 더 나을 수 있음

**질문**:
- 데모 질문 10개를 실제 데이터로 바꾸는 것이 3.5시간의 가치가 있는가?
- 심사위원이 "이 질문이 실제 데이터인지 가상인지" 구분할 수 있는가?
- 구분할 수 없다면 투자 대비 효과가 낮음

### 2. 데이터 접근 권한 문제
**문제**:
- Jira: 프로젝트별 권한 필요, 고객사 이슈는 비공개일 수 있음
- Slack: 채널 멤버가 아니면 메시지 읽기 불가 (이미 확인됨)
- Gmail: 개인 이메일 접근은 프라이버시 이슈
- Google Drive: 제안서/계약서는 기밀 문서일 가능성

**리스크**:
- 2.5시간 투자했는데 데이터 접근 실패 → 시간 낭비
- 일부만 접근 가능 → 불완전한 데이터

### 3. 데이터 품질 불확실성
**문제**:
- Jira 이슈: 제목만 있고 내용이 부실할 수 있음
- Slack 메시지: 맥락 없이 단편적일 수 있음
- Confluence: 최신 정보가 아닐 수 있음

**리스크**:
- 리서치했는데 사용 가능한 질문이 5개뿐 → 나머지는 결국 가상 데이터
- 실제 질문이 너무 전문적이거나 복잡해서 데모에 부적합

### 4. 데모 시나리오 제약
**문제**:
- 실제 질문은 우리가 원하는 흐름과 맞지 않을 수 있음
- 예: CHECK 시맨틱 분석을 보여주려면 복합 질문이 필요한데, 실제 데이터에 적절한 복합 질문이 없을 수 있음

**리스크**:
- 실제 데이터에 맞춰 데모 시나리오를 바꿔야 함 → 기존 설계와 충돌
- 또는 실제 데이터를 억지로 끼워 맞춤 → 부자연스러움

### 5. 개인정보/기밀 정보 노출 위험
**문제**:
- 실제 고객사 이름, 계약 내용, 기술 이슈가 데모에 노출
- 발표 영상이 녹화되고 공유될 수 있음

**리스크**:
- 고객사 동의 없이 정보 사용 → 법적/윤리적 문제
- 마크애니 내부 기밀 노출 → 보안 이슈

## 🤔 대안 제시

### 대안 1: 선택적 리서치 (1시간)
**방법**:
- Confluence FAQ만 리서치 (접근 쉬움, 공개 정보)
- TOP 5 질문만 추출
- 나머지 5개는 FAQ 기반으로 변형

**장점**:
- 시간 절약 (2.5h → 1h)
- 접근 권한 문제 최소화
- 기밀 정보 노출 위험 낮음

**단점**:
- 완전한 실제 데이터는 아님
- 차별화 효과 감소

### 대안 2: 시드 데이터 시뮬레이션 (30분)
**방법**:
- 실제 리서치 없이 "실제처럼 보이는" 질문 작성
- 발표에서 "실제 고객 질문 패턴 기반" (리서치 기반이 아닌 패턴 기반)
- 통계는 "추정치" 또는 "시뮬레이션"으로 명시

**장점**:
- 시간 최소화 (2.5h → 0.5h)
- 데모 시나리오에 완벽히 맞춤 가능
- 기밀 정보 문제 없음

**단점**:
- "실제 데이터 기반" 차별화 포인트 상실
- 심사위원이 "시뮬레이션"이라는 걸 알면 신뢰도 하락

### 대안 3: 하이브리드 접근 (1.5시간)
**방법**:
- Confluence FAQ 리서치 (30분) → 공개 정보, 안전
- 실제 질문 3개 + 변형 질문 7개
- 발표에서 "실제 FAQ 기반" (구체적 숫자는 언급 안 함)

**장점**:
- 시간 절약 (2.5h → 1.5h)
- 일부 실제 데이터 사용 → 신뢰도 확보
- 데모 시나리오 유연성 유지

**단점**:
- 완전한 실제 데이터는 아님

## 💡 추가 고려사항

### 1. 발표 전략 관점
**질문**: 심사위원이 "이 질문이 실제 데이터인가?"를 확인할 방법이 있는가?
- 없다면: 실제 데이터든 가상 데이터든 차이 없음 → 시간 투자 불필요
- 있다면: 실제 데이터 사용이 중요 → 시간 투자 가치 있음

### 2. 데모 완성도 vs 데이터 진정성
**트레이드오프**:
- 3.5시간을 데이터 리서치에 쓰면 → 데이터 진정성 ↑, 기능 완성도 ↓
- 3.5시간을 기능 구현에 쓰면 → 기능 완성도 ↑, 데이터 진정성 ↓

**해커톤 심사 기준**:
- 완성도 > 데이터 진정성 (일반적)
- 작동하는 데모 > 실제 데이터

### 3. 작년 1등팀과의 비교
**작년 1등팀**: 캘린더 자동 회의실 잡기
- 실제 데이터 사용 여부: 불명
- 승리 요인: 아이디어 + 완성도 (데이터 진정성 아님)

**우리 차별화 포인트**:
- 실제 데이터보다 "실적용 가능성"이 더 중요
- 실적용 가능성은 데모 완성도로 증명

## 🎯 권장 사항

### 최종 추천: 대안 3 (하이브리드 접근)

**이유**:
1. **시간 효율**: 1.5시간 투자 (2.5h 대비 40% 절감)
2. **리스크 관리**: Confluence만 사용 → 접근 권한 문제 최소화
3. **신뢰도 확보**: 일부 실제 FAQ 사용 → "실제 기반" 주장 가능
4. **유연성**: 데모 시나리오에 맞게 조정 가능
5. **기밀 보호**: 공개 FAQ만 사용 → 기밀 정보 노출 없음

**구체적 실행 계획**:
1. Confluence "제품 FAQ" 페이지 리서치 (30분)
2. TOP 10 질문 추출 (10분)
3. 데모 시나리오에 맞게 3개 선택 + 7개 변형 (30분)
4. 답변 데이터 준비 (20분)
5. **총 1.5시간**

**발표 표현**:
- ✅ "실제 고객 FAQ 기반"
- ✅ "자주 묻는 질문 패턴 반영"
- ❌ "Jira 이슈 1,234건 분석" (과장, 검증 불가)
