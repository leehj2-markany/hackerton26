## 1. 핵심 아이디어: 컴파일러 비유

CHECK의 핵심 통찰은 **"컴파일러와 LLM 추론의 유사성"**이다.

- 소스코드는 실행 전에 **시맨틱 분석(타입 체킹)**을 거쳐 오류를 제거한다
- LLM의 추론 체인도 실행 전에 **시맨틱 일관성 검사**를 거쳐야 한다
- 즉, "LLM이 생각하는 과정을 실행하기 전에 먼저 검증하자"

### 비유 구조
```
[소스코드] → [시맨틱 분석/타입체킹] → [컴파일] → [실행] → [결과]
[멀티홉 질문] → [추론 체인 타입체킹] → [체인 정렬] → [순차 실행] → [답변]
```

---

## 2. CHECK 프레임워크 3단계

### 2.1 타입 추출 (Type Extraction)
- 모든 엔티티를 **Person / Place / Thing** 3가지 타입으로 분류
- 관계(relationship)도 입력/출력 타입을 가진 **함수**로 모델링
  - 예: `born_in(Person) → Place`, `creator_of(Thing) → Person`
- 엔티티 타입: ReFinED 엔티티 링킹 모델 + LLM 조합으로 추출
- 관계 타입: 사전에 템플릿 라이브러리로 구축 (dense retrieval로 매칭)

### 2.2 질문 분해 (Question Decomposition)

#### Chain Extraction
- 멀티홉 질문 → LLM으로 관계 체인 추출
- 예: "현재 미국 대통령의 배우자는?" → `[배우자, 대통령]`
- 역순 추출 (masked attention 방향과 일치시켜 정확도 향상)

#### Chain Alignment (타입 체킹)
- 추출된 체인의 각 관계에 입력/출력 타입 부여
- 인접 관계 간 타입 일관성 검사: `출력 타입(i) == 입력 타입(i+1)`?
- **정렬 페널티(A)** 계산: 불일치 수 합산
- A=0이면 정렬 완료, A>0이면 수리 필요

#### Chain Repair (체인 수리)
- **순열 최적화**: 관계 순서를 재배열하여 A=0인 조합 탐색
- 여러 A=0 조합이 있으면 **최소 순열 비용(λ)** 선택
- 순열로 해결 안 되면 → **LLM 온도를 0.1씩 올려 재추출** (0.0→1.0)
- 최종 실패 시 → 가장 낮은 A를 가진 체인 사용

### 2.3 서브질문 해결 (Subquestion Resolution)
- 정렬된 체인을 순차 순회하며 각 홉 답변
- 각 (엔티티, 관계) 쌍을 편집 저장소와 코사인 유사도 비교
- 유사도 > 0.8 → 편집된 답변 사용
- 유사도 ≤ 0.8 → LLM에게 질문 생성 후 답변 요청
- 최종 엔티티까지 반복

## 3. 실험 결과

### 성능
- **MQuAKE 데이터셋** 4개 서브셋, **3개 LLM** (GPT-J, Vicuna-7B, Falcon-7B)
- 기존 SOTA 대비 **평균 22.8% 정확도 향상**
- 세부: MQuAKE-CF-3k +31.57%, MQuAKE-2002 +28.51%, MQuAKE-Hard +24.79%, MQuAKE-T +16.77%

### 주요 발견
- 홉 수 증가 → 정확도 감소 (체인 길이 증가로 오류 누적)
- 편집 수 증가 → 정확도 감소 (더 복잡한 질문)
- 온도 증가로 재추출 시 추가 정답 확보 가능 (0이 아닌 정확도)
- 코사인 유사도가 dot product보다 편집 매칭에 효과적
- 최적 임계값: 0.80

## 4. CHECK의 기술적 혁신 포인트

| 혁신 | 설명 |
|------|------|
| 시맨틱 타입 체킹 | 추론 체인의 논리적 일관성을 사전 검증 |

## 5. ANY 브릿지 적용 가능성 분석

### 5.1 직접 적용 가능한 아이디어

#### ① 질문 분해 + 라우팅 강화 (R3 Agentic 라우팅 보강)
현재 R3의 Agentic 라우터는 인텐트 분류(기술질문/가격문의/일반/불명확)만 수행.
CHECK의 **질문 분해** 아이디어를 적용하면:

```
고객: "DRM 제품의 윈도우 11 호환성과 도입 절차를 알려주세요"
→ 분해: [기술질문: DRM 윈도우 11 호환성] + [프로세스 질문: DRM 도입 절차]
→ 각각 다른 File Search Store / 처리 경로로 라우팅
→ 종합 답변 생성
```

이미 요구사항에 "복합 질문 분해"가 Agentic RAG 개념으로 포함되어 있으나,
CHECK의 **타입 체킹으로 분해 품질을 검증**하는 아이디어는 추가 가치가 있다.

#### ② 추론 과정 시각화 (R11 AI 투명성 UI 보강)
CHECK의 "체인 추출 → 타입 체킹 → 순차 실행" 과정은
R11의 "AI 사고 과정" 패널과 자연스럽게 연결:

```
🤔 질문 분석 중...
→ 질문 분해: [DRM 호환성](기술) + [도입 절차](프로세스)
→ 타입 검증: ✅ 논리적 일관성 확인
→ 정책기능서 검색 중... 3건 발견
→ 답변 생성 중...
```

#### ③ 신뢰도 평가 강화 (R9 AI 안전성 보강)
CHECK의 **정렬 페널티(A)** 개념을 신뢰도 평가에 활용:
- A=0 (완벽 정렬) → 🟢 높은 신뢰도
- A>0 (부분 정렬) → 🟡 중간 신뢰도 + "정확하지 않을 수 있습니다"
- 체인 수리 실패 → 🔴 낮은 신뢰도 → 담당자 연결 권유

### 5.2 해커톤 MVP에서의 현실적 적용 범위

| CHECK 아이디어 | 적용 가능성 | 구현 난이도 | 권장 |
|---------------|-----------|-----------|------|
| 질문 분해 (복합 질문 → 서브질문) | ✅ 높음 | 중 (프롬프트 엔지니어링) | Tier 1 R3에 반영 |
| 타입 체킹 (추론 일관성 검증) | 🟡 중간 | 높음 (타입 라이브러리 필요) | Tier 3 또는 향후 |
| 순열 기반 체인 수리 | ❌ 낮음 | 매우 높음 | 향후 실서비스 |
| 온도 스케일링 재추출 | 🟡 중간 | 낮음 (API 파라미터) | Tier 2 fallback |
| SR 임베딩 + 코사인 유사도 | ❌ 해당없음 | - | Gemini File Search가 대체 |

### 5.3 결론: 현재 요구사항에 이미 반영된 부분 vs 추가 반영 가능한 부분

**이미 반영된 것:**
- 질문 분해 → R3 Agentic RAG의 "복합 질문 하위 질문 분해"
- 추론 과정 시각화 → R11 AI 투명성 UI의 "AI 사고 과정 패널"
- 신뢰도 기반 라우팅 → R3의 ConfRAG 패턴 (🟢🟡🔴)

**추가 반영 가능한 것:**
- 분해된 서브질문의 **논리적 일관성 검증** (타입 체킹 간소화 버전)
- 답변 실패 시 **온도 올려 재시도** (fallback 전략)
- 정렬 페널티를 **신뢰도 점수에 반영** (정량적 신뢰도)

**해커톤 51시간 내 권장 사항:**
CHECK의 핵심 아이디어(질문 분해 + 일관성 검증)는 이미 R3 Agentic RAG와 R11 AI 투명성 UI에 
개념적으로 반영되어 있다. 추가로 "타입 체킹" 수준의 정교한 검증을 구현하기에는 51시간이 부족하므로,
**프롬프트 엔지니어링 수준에서 "분해 결과 검증" 단계를 추가**하는 것이 현실적이다.

## 6. 한 줄 요약

> CHECK는 "LLM의 추론 과정을 컴파일러처럼 타입 체킹하여 논리적 오류를 사전에 잡는" 프레임워크로,
> 멀티홉 QA 정확도를 평균 22.8% 향상시켰다. ANY 브릿지에는 질문 분해와 신뢰도 평가 강화에 
> 부분적으로 적용 가능하며, 핵심 아이디어는 이미 R3/R11에 개념적으로 반영되어 있다.
