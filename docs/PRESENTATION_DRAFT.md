### 슬라이드 3: 배경 — Customer First (90초)

**"Customer First" — 고객 접점에서 발견한 3가지 문제**

| # | 문제 | 현상 | 영향 |
|---|------|------|------|
| 1 | 인바운드 리드 응답이 너무 느림 | 고객 홈페이지 문의 → 담당자 연결까지 24~48시간 | 영업 기회 누락, 고객 이탈 |
| 2 | 기고객 관리가 안 됨 | 기고객 매출 비중이 큰데, 유지보수 문의 대응이 수동 | 기고객 만족도 하락, 이탈 리스크 |
| 3 | 기술 자산이 파편화됨 | MTG↔MCG 간 기술 문서/구축 경험이 조직 자산으로 축적 안 됨 | 신규 프로젝트마다 처음부터 시작 |

**데이터 기반 근거 — Slack 130개+ 채널 분석 (Slack MCP 활용):**

| 발견한 패턴 | 증거 |
|-----------|------|
| 제품별 기술지원 채널 난립 | `hotline-drm`, `hotline-dlp`, `hotline-tracer` 등 5개+ hotline 채널 |
| 프로젝트 지식 사일로 | 50개+ 고객사별 프로젝트 채널에 구축 경험 분산 |
| 영업/제안 프로세스 수동 | `0-markany-csteam`에서 수동 멘션으로 문의 이관 |
| 기술지원 Q&A 반복 | 이미 워크플로우("질문 생성 수집")로 구조화 시도 → 근본 해결 안 됨 |
| 파트너 기술지원 반복 | 협력사별 채널에 동일 제품 자료 반복 공유 |

> 발표 멘트: "Customer First. 저희가 발견한 문제는 세 가지입니다. 첫째, 고객이 홈페이지에서 문의하면 담당자 연결까지 1~2일이 걸립니다. 둘째, 기고객 매출 비중이 큰데 유지보수 문의 대응이 수동입니다. 셋째, 기술 자산이 조직별로 파편화되어 축적이 안 됩니다. 이건 추측이 아닙니다. Slack MCP로 전사 워크스페이스 130개 이상의 채널을 분석해서 데이터로 확인한 결과입니다. 제품별 hotline 채널이 5개 이상 운영되고 있었고, 이미 워크플로우로 질문을 구조화하려는 시도가 있었지만 근본적인 해결은 안 된 상태였습니다."

### 슬라이드 4: 배경 — As-Is 현황 (60초)

**현재 고객 문의 프로세스 (As-Is)**

```
고객 홈페이지 문의 → CS팀 수동 확인 → Slack 멘션으로 이관 → 담당자 확인 → 회신
                    (수시간)          (수시간)              (수시간~1일)
                                     총 24~48시간
```

**페인 포인트:**
- 고객: 문의 후 언제 답변이 올지 모름 → 불안, 이탈
- CS팀: 28개 제품 중 어떤 제품인지 파악 → 적합한 담당자 찾기 → 수동 멘션
- 담당자: Slack 알림 놓침 → 응답 지연 → 고객 불만
- 조직: 해결된 문의의 지식이 축적되지 않음 → 같은 질문 반복

**핵심 질문:**
> "고객이 홈페이지에서 질문하면, AI가 즉시 답변하고,
> 필요하면 담당자까지 자동으로 연결해줄 수 없을까?
> 그리고 그 과정에서 지식이 자동으로 쌓이면?"

> 발표 멘트: "현재 프로세스를 보시면, 고객 문의부터 담당자 회신까지 24~48시간이 걸립니다. CS팀은 28개 제품 중 어떤 건지 파악하고 적합한 담당자를 찾아 수동으로 멘션해야 합니다. 담당자가 Slack 알림을 놓치면 응답은 더 지연됩니다. 그리고 해결된 문의의 지식은 어디에도 축적되지 않습니다."

### 슬라이드 5: 서비스 소개 — ANY 브릿지 3단계 구조 (90초)

**ANY 브릿지 = AI 프리세일즈 어시스턴트**

```
┌─────────────────────────────────────────────────────────────┐
│  ① AI 챗봇 (즉시, 평균 8초)                                  │
│     고객 질문 → 28개 제품 지식 기반 RAG 즉시 답변              │
│     고객명 입력 → 유지보수 현황 기반 담당자 자동 매칭           │
│                                                             │
│         70% 여기서 해결 ✅                                    │
│         ↓ (AI 해결 불가 시)                                   │
│                                                             │
│  ② 담당자 연결 (스마트 에스컬레이션)                           │
│     AI가 질문 분해 → 서브질문별 적합한 담당자 자동 배정         │
│     Slack 채널 자동 생성 → 담당자 실시간 답변                  │
│     담당자 답변 → 고객 챗봇에 LIVE 전달                       │
│                                                             │
│         30% 여기서 해결 ✅                                    │
│         ↓                                                    │
│                                                             │
│  ③ 완료 & 축적                                               │
│     해결 완료 → 만족도 평가 → 대화 이력 DB 축적               │
│     → AI 학습 데이터로 고도화 (선순환)                         │
└─────────────────────────────────────────────────────────────┘
```

**핵심 컨셉: AI가 문을 열고, 인간이 해결하고, 지식이 쌓인다**

> 발표 멘트: "ANY 브릿지는 3단계 구조입니다. 1단계, AI 챗봇이 고객 질문에 즉시 답변합니다. 28개 제품 기술 문서를 학습한 RAG 기반이고, 평균 8초 안에 답변합니다. 약 70%의 문의가 여기서 해결됩니다. 2단계, AI가 해결하기 어려운 문의는 질문을 분해해서 적합한 담당자에게 자동 배정합니다. Slack 채널이 자동 생성되고, 담당자 답변이 고객 챗봇에 실시간으로 전달됩니다. 3단계, 모든 대화 이력이 DB에 축적되어 AI가 점점 똑똑해지는 선순환 구조입니다."

### 슬라이드 6: 핵심기능 ① — AI 즉시 답변 + 고객 매칭 (60초)

**데이터 소스 — 이미 있는 자산을 활용**

| 데이터 | 출처 | 용도 |
|--------|------|------|
| 정책기능서 (28개 제품) | Google Sheets (박희수 본부장 정제) | RAG 지식 베이스 |
| 하이퍼링크 2depth 문서 17건 | IST 표준기능정의서, 모듈담당자, 스펙정의서 등 | 상세 기술 정보 |
| 유지보수 현황 시트 | 세실님 관리 시트 | 고객-담당자 자동 매칭 |
| 고객 미팅 녹취 35건+ | Slack 채널 | 테스트 쿼리 50개 추출, RAG 품질 검증 |

**AI가 하는 일:**
- 고객 매칭: 회사명 입력 → 유지보수 시트 기반 담당 영업/SE/개발자 자동 식별
- 즉시 답변: 54개 지식 청크 기반 시맨틱 검색 (pgvector) + 자연어 답변 생성
- 복합질문 분해: "DRM과 개인정보추출 250유저 구축검토" → 서브질문별 독립 RAG → 통합 답변
- 기능 중복 제품 구분: "비가시성 워터마크" → Print TRACER(출력물) + Screen TRACER(화면) 자동 구분 안내

> 발표 멘트: "ANY 브릿지는 새로 만든 데이터가 아니라, 이미 있는 자산을 활용합니다. 박희수 본부장님이 정제한 28개 제품 정책기능서, 세실님의 유지보수 현황 시트, 그리고 실제 고객 미팅 녹취 데이터입니다. 흩어져 있던 자산을 AI가 연결한 겁니다."

### 슬라이드 7: 핵심기능 ② — 스마트 에스컬레이션 + 지식 축적 (60초)

**에스컬레이션 흐름:**

```
고객: "견적과 자료를 받고 싶네요"
  ↓
AI Router: 질문 분해 + 담당자 배정
  → "DRM 제품 구성" → 이현진 (SE)
  → "250유저 견적" → 송인찬 (영업)
  ↓
Slack 채널 자동 생성: #anybridge-sk하이닉스-drm-구축검토
  → 채소희(고객센터) 입장 → 질문 분류 안내
  → 이현진, 송인찬 순차 입장 → 각자 담당 질문 답변
  ↓
담당자 답변 → 고객 챗봇에 LIVE 표시
  ↓
해결 완료 → 대화 이력 DB 축적 → AI 고도화 데이터
```

**차별점:**
- 일반 챗봇: AI 답변 실패 → "담당자에게 문의하세요" (끝)
- ANY 브릿지: AI 답변 실패 → 질문 분해 → 적합한 담당자 자동 배정 → 실시간 협업 → 지식 축적

> 발표 멘트: "일반 챗봇은 AI가 답변 못 하면 '담당자에게 문의하세요'로 끝납니다. ANY 브릿지는 다릅니다. AI가 질문을 분해하고, 서브질문별로 적합한 담당자를 자동 배정합니다. Slack 채널이 자동 생성되고, 담당자가 답변하면 고객 챗봇에 실시간으로 전달됩니다. 그리고 이 모든 대화가 축적되어 AI가 점점 똑똑해집니다."

### 슬라이드 9: 애플리케이션 스택 (60초)

| 레이어 | 기술 | 선택 이유 |
|--------|------|----------|
| Frontend | React 18 + Vite 5 + Tailwind 3 | 빠른 빌드, 홈페이지 임베딩 용이 |
| Backend | Vercel Serverless (Node.js ES modules) | 서버리스 무중단 배포, 콜드스타트 최소 |
| LLM | Gemini 2.5 Flash/Pro + Claude Opus 4 | 3-Tier 비용 최적화 (아래 상세) |
| Vector DB | Supabase PostgreSQL + pgvector | 시맨틱 검색 + RDB 통합, 별도 벡터 DB 불필요 |
| Embedding | Gemini gemini-embedding-001 (3072차원) | Gemini 생태계 통일 |
| Messaging | Slack Bot API (xoxb) | 사내 Slack 직접 연동 |
| Deployment | Vercel (Hobby) | 프론트+백엔드 단일 배포 |
| MCP 연동 | Slack, Confluence, Jira, Google Workspace, Salesforce | Kiro IDE에서 데이터 수집 자동화 |

**3-Tier LLM 라우팅 — 비용 vs 품질 최적화:**

| 복잡도 | 모델 | 비율 | 용도 |

### 슬라이드 10: AI Agent 기술 ① — LLM-as-a-Router + DESV 파이프라인 (90초)

**LLM-as-a-Router: 왜 키워드 룰을 버렸나?**

```
❌ 키워드 룰: "견적" → complex  |  "그래서 어쩌라는거죠?" → complex (오판!)
✅ LLM Router: Flash 1회 호출로 복잡도 + 서브질문 + 담당자 + 고객조건 동시 추출
```

Router 1회 호출 결과 예시:
```json
{
  "complexity": "complex",
  "subQuestions": [
    { "question": "DRM 제품 구성은?", "assignee": "이현진" },
    { "question": "250유저 견적은?", "assignee": "송인찬" }
  ],
  "extractedContext": {
    "product": "Document SAFER", "userScale": "250", "intent": "구축검토"
  }
}
```

**DESV 파이프라인 (CHECK 논문 기반) — 복합질문 전용:**

```
Phase 1: Decompose (분해) → 서브질문 N개로 분해
Phase 2: Enrich (강화)    → 서브질문별 독립 RAG 검색 (LLM 추가 호출 0회)
Phase 3: Synthesize (통합) → 구조화 프롬프트로 통합 답변 생성 (Flash 1회)
Phase 4: Verify (검증)    → 각 서브질문 커버리지 확인
```

결과: 단일 RAG 대비 복합질문 답변 커버리지 대폭 향상, 추가 LLM 비용 0

> 발표 멘트: "처음에는 키워드 룰로 복잡도를 판단했는데, '그래서 어쩌라는거죠?'를 complex로 오판하는 등 엣지케이스를 못 잡았습니다. 그래서 LLM에 판단을 위임했습니다. Flash 1회 호출로 복잡도, 서브질문 분해, 담당자 배정, 고객 조건 추출을 동시에 수행합니다. 복합 질문은 DESV 파이프라인으로 처리합니다. CHECK 논문 기반으로 설계했고, 서브질문별 독립 RAG 검색 후 통합 답변을 생성하고, 모든 서브질문이 커버되었는지 검증합니다."

**Q1: "실제 고객이 쓸 수 있는 수준인가요?"**
> A: 네, 이미 라이브 배포되어 있고 (hackerton-kappa.vercel.app), 28개 제품 정보가 학습되어 있습니다. 실제 고객 미팅 녹취 50개 쿼리로 테스트해서 94% 정확도를 달성했습니다. Slack 에스컬레이션도 실제 동작합니다.

**Q2: "AI가 틀린 답변을 하면 어떻게 되나요?"**
> A: 4중 안전장치가 있습니다. 입력 인젝션 방어, PII 마스킹, Constitutional AI 출력 검증, 그리고 AI가 실패하면 자동으로 담당자 연결로 전환하는 Graceful Degradation입니다. AI가 확신이 낮으면 스스로 에스컬레이션을 제안합니다.

**Q3: "Gemini와 Claude를 왜 같이 쓰나요?"**
> A: 3-Tier 전략입니다. 70%의 단순 질문은 Gemini Flash로 빠르고 저렴하게, 복합 질문은 Flash + DESV 파이프라인으로, 안전성 검증만 Claude Opus 4를 사용합니다. 단일 모델 대비 비용 70% 절감하면서 품질은 유지합니다.

**Q4: "Kiro IDE가 없었으면 어떻게 달라졌을까요?"**
> A: 가장 큰 차이는 MCP 연동입니다. Slack 130개 채널 분석, Google Sheets 제품 정보 수집, Confluence 문서 참조를 IDE 안에서 바로 할 수 있었습니다. 이걸 수동으로 했으면 데이터 수집만 일주일은 걸렸을 겁니다. 그리고 전문가 페르소나 리뷰로 혼자 개발하면서도 다각도 검증이 가능했습니다.

**Q5: "작년 해커톤 1등과 뭐가 다른가요?"**
> A: 작년 1등은 캘린더 자동 회의실 잡기였는데, 내부 편의 도구였습니다. ANY 브릿지는 고객 접점 도구입니다. 고객 문의 응답 시간을 24~48시간에서 8초로 줄이고, 그 과정에서 기술 자산이 자동 축적됩니다. 실사용 가능성과 비즈니스 임팩트가 다릅니다.

**Q6: "비용은 얼마나 드나요?"**
> A: Vercel Hobby (무료), Supabase Free tier, Gemini Flash (1000 쿼리당 약 $0.5), Claude Opus는 안전성 검증에만 사용 (전체의 5%). 월 운영비 $10~20 수준으로 추정됩니다.

**Q7: "카카오톡 연동은 가능한가요?"**
> A: 현재는 Slack 기반이지만, 에스컬레이션 모듈이 추상화되어 있어서 카카오톡 API로 교체하면 됩니다. Phase 2에서 카카오톡 채널 확장을 계획하고 있습니다.
