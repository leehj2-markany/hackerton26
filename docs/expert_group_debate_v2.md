## 📚 AI 전문가 그룹 리서치 결과 (최신 논문/기술 기반)

### 리서치 1: Gemini File Search API — 매니지드 RAG (2025.11 출시)

**출처:** [Google Gemini File Search](https://www.philschmid.de/gemini-file-search-javascript), [DataCamp Tutorial](https://www.datacamp.com/tutorial/google-file-search-tool)

**핵심 발견:**
- Gemini API에 내장된 완전 관리형 RAG 시스템
- 문서 업로드만 하면 자동으로 chunking → embedding → vector indexing → semantic search 처리
- 별도 벡터 DB(Pinecone, Weaviate 등) 불필요
- presalesapp_in_slack의 기존 수동 RAG 파이프라인(rag.js)을 이것으로 교체하면 개발 시간 대폭 절감

**ANY 브릿지 적용 임팩트:**
- 기존: 정책기능서 → 수동 chunking → 임베딩 생성 → 벡터 DB 저장 → 검색 (rag.js 16KB 코드)
- 변경: 정책기능서 → Gemini File Search Store 업로드 → 끝
- **예상 절감 시간: 8~12시간** (RAG 파이프라인 구축 시간 거의 제거)

### 리서치 2: ConfRAG — 신뢰도 기반 선택적 RAG (2025 논문)

**출처:** [ConfRAG - arxiv](https://arxiv.org/html/2506.07309)

**핵심 발견:**
- ConfQA: 모델이 확신 없으면 "I am unsure"로 응답하도록 학습
- ConfRAG: 모델이 "unsure" 응답 시에만 RAG 검색 트리거
- 환각률 20~40% → 5% 이하로 감소
- 불필요한 외부 검색 30% 이상 감소

**ANY 브릿지 적용:**
- Gemini에 직접 ConfQA 파인튜닝은 불가하지만, **프롬프트 엔지니어링으로 유사 효과 구현 가능**
- 시스템 프롬프트: "정책기능서에서 근거를 찾을 수 없으면 반드시 '확인이 필요한 내용입니다'로 응답하라"
- confidence 3단계: 높음(직접 답변) / 중간(답변 + 면책) / 낮음(담당자 연결 권유)

### 리서치 3: CRAG (Corrective RAG) — Google DeepMind (2024)

**출처:** [CRAG Paper](https://www.aimodels.fyi/papers/arxiv/corrective-retrieval-augmented-generation)

**핵심 발견:**
- 검색된 문서의 품질을 평가하는 경량 retrieval evaluator 도입
- 문서 품질에 따라 3가지 액션: Correct(사용) / Ambiguous(보완 검색) / Incorrect(웹 검색 fallback)
- 기존 RAG 대비 정확도 유의미하게 향상

**ANY 브릿지 적용:**
- 정책기능서 검색 결과의 relevance score 평가
- score 높음 → 직접 답변
- score 중간 → 추가 문서 검색 후 답변
- score 낮음 → "정확한 답변을 위해 담당자에게 연결합니다" (에스컬레이션 자동 트리거)
- **이 로직이 3단계 구조(AI → 담당자 → 축적)의 1→2단계 전환을 자동화**

### 리서치 4: Self-RAG — 자기 성찰형 RAG (ICLR 2024)

**출처:** [Self-RAG - arxiv](https://arxiv.org/abs/2310.11511), [OpenReview](https://openreview.net/forum?id=hSyW5go0v8)

**핵심 발견:**
- LLM이 스스로 "검색이 필요한가?" 판단 → 필요 시에만 검색
- 생성 후 자기 비평(self-critique) → 품질 낮으면 재생성
- reflection token으로 답변 품질 자체 평가

**ANY 브릿지 적용:**
- Gemini 프롬프트에 self-reflection 단계 추가:
  1. 질문 분석 → "이 질문에 검색이 필요한가?"
  2. 검색 결과 평가 → "이 문서가 질문에 관련 있는가?"
  3. 답변 생성 후 → "이 답변이 문서 근거에 충실한가?"
- 각 단계를 로그로 남기면 → 대시보드에서 AI 의사결정 과정 시각화 가능

### 리서치 5: RAGAS 프레임워크 — RAG 품질 평가 (2025)

**출처:** [RAGAS Framework](https://www.emergentmind.com/topics/ragas-framework), [GeeksforGeeks](https://www.geeksforgeeks.org/artificial-intelligence/ragas/)

**핵심 발견:**
- RAG 시스템 평가를 위한 4대 메트릭:
  - Faithfulness: 답변이 검색된 문서에 충실한가?
  - Answer Relevancy: 답변이 질문에 관련 있는가?
  - Context Precision: 검색된 문서가 질문에 관련 있는가?
  - Context Recall: 필요한 정보가 모두 검색되었는가?
- ground truth 없이도 평가 가능 (reference-free)

**ANY 브릿지 적용:**
- 벤치마크 테스트 시 RAGAS 메트릭으로 AI 품질 정량 측정
- 발표에서 "Faithfulness 92%, Answer Relevancy 88%" 같은 수치 제시 → 신뢰도 극대화

### 리서치 6: Agentic RAG — 에이전틱 RAG (2025 트렌드)

**출처:** [Agentic RAG](https://www.emergentmind.com/topics/agentic-retrieval-augmented-generation-rag), [Enterprise Guide](https://blog.dedicatted.com/insights/agentic-rag-architecture-patterns-that-actually-work-for-enterprise-ai)

**핵심 발견:**
- 기존 RAG: 검색 → 생성 (단순 파이프라인)
- Agentic RAG: 계획 → 검색 → 평가 → 도구 사용 → 재검색 → 생성 (자율 에이전트)
- 복잡한 질문을 하위 질문으로 분해 → 각각 검색 → 종합 답변
- 도구 통합: 검색 외에 계산, API 호출, DB 조회 등 가능

**ANY 브릿지 적용:**
- 고객 질문이 복합적일 때: "DRM 제품의 윈도우 11 호환성과 가격을 알려주세요"
  → 하위 질문 1: "DRM 윈도우 11 호환성" (정책기능서 검색)
  → 하위 질문 2: "DRM 가격" (가격은 정책기능서에 없음 → 담당자 연결)
  → 종합: "호환성은 [답변], 가격은 담당자에게 확인이 필요합니다"
- **51시간이면 이 수준까지 구현 가능**

### 리서치 7: 2026 RAG 고객지원 벤치마크 리포트

### 라운드 2: 기존 전문가 그룹 반응 + 상호 소통

**카파시 → 제리 리우에게:**

> 이 구조를 Gemini File Search로 옮기면 제품별 필터링 같은 세밀한 제어가 가능한가요?

**제리 리우 → 카파시에게:**
> 좋은 지적입니다. Gemini File Search는 store 단위로 분리 가능합니다.
> 제품별로 별도 store를 만들면 됩니다: store_drm, store_safecopy, store_document_safer 등.
> 질문이 들어오면 먼저 제품 분류 → 해당 store에서만 검색.
> 이게 해리슨이 말한 Agentic RAG의 라우팅과 자연스럽게 결합됩니다.
> 단, store당 쿼리 5개 제한이 있으니 store 수는 5개 이하로 유지해야 합니다.

**해커톤 탑티어 → AI 전문가 그룹에게:**
> 기술적으로 훌륭한데, **데모에서 이걸 어떻게 보여줄 건가요?**

> 예: 챗봇 답변 옆에 작은 배지 — "📚 정책기능서 3건 참조" 또는 "⚠️ 확신도 낮음"

> 세일즈포스 데이터로 현재 baseline을 잡고, ANY 브릿지 적용 후 개선치를 보여주면 킬링입니다.
> 팀 내에서 세일즈포스 접근 권한자에게 데이터 요청을 병행해야 합니다.

> 동의합니다. 그리고 KPI를 더 확장하면:

> 5. 재문의률 (같은 고객이 같은 질문 반복) — 낮을수록 1차 해결 품질 높은 것
> 6. 영업 기회 전환율 (문의 → 영업 기회 감지) — 이건 MBA님이 제안한 "영업 기회 자동 감지"와 연결
>
> 그리고 **케이스 상태 머신**이 이 모든 KPI의 기반입니다.
> 케이스 없이는 메트릭 측정 자체가 불가능합니다.

**카파시 → MBA + 세일즈포스에게:**
> KPI 6개는 너무 많습니다. 51시간 안에 6개 KPI를 다 측정하고 대시보드에 넣으면
> 대시보드가 복잡해져서 오히려 임팩트가 떨어집니다.
> **핵심 3개만 대시보드에, 나머지는 발표 슬라이드에서 언급**하는 게 맞습니다.
> 대시보드: 1차 해결률, 평균 응답 시간, 만족도
> 발표: 에스컬레이션률, 재문의률, 영업 기회 (숫자만 언급)

> - google-drive-client.js → 제거 (직접 업로드)

**AWS 프로덕트 오피서 → 제리 리우에게:**

**해리슨 체이스 (최종 정리):**
> Agentic RAG 라우팅 최종 권고:
> 1. 질문 수신 → **인텐트 분류** (기술질문 / 가격문의 / 일반문의 / 불명확)

> 4. 일반문의 → 전체 Store 검색 → 답변 생성
> 5. 불명확 → "질문을 좀 더 구체적으로 해주시겠어요?" 재질문

### 답변 범위 제한 MVP 제외 결정

**사유:** 해커톤 데모 중 정상적인 질문이 차단되는 역효과 리스크.
- 예: "마크애니 회사 소개" → "제품 관련 질문만 답변 가능" (어색)
- 환각 방지(ConfRAG 패턴)가 상위 호환으로 충분히 커버
- 근거 없는 질문 → "확인이 필요한 내용입니다" (자연스러운 대응)

AI 안전성 프레임워크: 4종 → **3종**으로 조정
1. ✅ 프롬프트 인젝션 차단
2. ✅ 환각 방지 (ConfRAG 패턴)
3. ✅ 개인정보 필터링
4. ❌ ~~답변 범위 제한~~ → 향후 실서비스 적용 시 검토
