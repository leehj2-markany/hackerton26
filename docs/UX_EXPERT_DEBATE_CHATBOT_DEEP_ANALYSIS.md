> 범위: 챗봇의 모든 기능 — AI 대화, 스트리밍, ThinkingPanel, 에스컬레이션, Slack 연동, 세션 관리, 반응형 UI

## 1. ThinkingPanel 위치 문제 (Critical)

**스크린샷 기반 관찰**: AI 답변이 상단에 먼저 표시되고, ThinkingPanel("AI 분석 중")이 하단에 위치

**김도현**: 이건 정보 계층(Information Hierarchy) 역전이다. 사용자가 질문을 보내면 자연스러운 기대는 "분석 → 답변" 순서인데, 현재는 답변 텍스트가 스트리밍으로 먼저 올라오고 ThinkingPanel이 그 아래에 깔린다. 사용자 시선이 위아래로 분산된다.

**박서연**: 맞다. ChatGPT, Claude 등 주요 AI 챗봇은 thinking indicator를 답변 위에 놓거나, 답변 자체에 통합한다. 현재 구조는 `messages.map()` 이후에 `{showThinking && <ThinkingPanel />}`이 렌더링되니까 항상 메시지 목록 아래에 위치한다. 스트리밍 답변과 ThinkingPanel이 동시에 보이면 사용자는 "이미 답변이 나오고 있는데 왜 아직 분석 중이지?"라고 혼란스러워한다.

**정민수**: B2B 고객은 특히 "이 AI가 제대로 분석하고 있나?"를 중요하게 본다. ThinkingPanel은 신뢰 구축 장치인데, 답변 아래에 묻히면 그 역할을 못 한다.

### 제안
- **Option A**: ThinkingPanel을 답변 메시지 위에 삽입 — 스트리밍 시작 전에 ThinkingPanel을 메시지로 추가하고, 스트리밍 완료 시 ThinkingPanel을 접거나 제거
- **Option B**: ThinkingPanel을 답변 버블 내부에 통합 — 답변 상단에 접을 수 있는 "분석 과정" 섹션으로 표시 (현재 `CollapsibleAIMessage`의 `thinkingProcess` 버튼과 유사하지만, 실시간으로)
- **Option C**: ThinkingPanel을 채팅 영역 상단에 sticky로 고정 — 스크롤과 무관하게 항상 보이게

**합의**: Option A가 가장 자연스럽다. 분석 → 답변 순서가 시간순으로 맞고, 완료 후 자동 접기하면 공간도 절약된다.

## 2. 스트리밍 답변 UX (High)

**박서연**: 스트리밍 구현 자체는 좋은데, 몇 가지 문제가 있다:

### 2-1. 스트리밍 중 커서/인디케이터 없음
현재 `isStreaming: true`인 메시지에 시각적 구분이 없다. 텍스트가 그냥 늘어나기만 하면 사용자는 "아직 생성 중인지, 끝난 건지" 구분이 안 된다.

**이준호**: 간단하다. 스트리밍 중인 메시지 끝에 깜빡이는 커서(`▌`)를 추가하면 된다. CSS `animation: blink`으로 처리.

### 2-2. 스트리밍 중 스크롤 문제
`useEffect(() => scrollToBottom(), [messages])`인데, 스트리밍 중에는 `messages` 배열이 변경되지 않고 기존 메시지의 `text` 속성만 업데이트된다. `setMessages(prev => prev.map(...))`이므로 배열 참조는 바뀌지만, 긴 답변에서 스크롤이 따라가지 않을 수 있다.

### 2-3. 스트리밍 중 마크다운 렌더링 깜빡임
`ReactMarkdown`이 매 토큰마다 전체 텍스트를 다시 파싱한다. 짧은 답변은 괜찮지만, 긴 답변(500자+)에서는 렌더링 비용이 누적된다.

**이준호**: 스트리밍 중에는 `ReactMarkdown` 대신 `<pre>` 또는 plain text로 표시하고, 스트리밍 완료 후 마크다운으로 전환하는 방식이 성능상 낫다. 또는 `useMemo`로 마크다운 파싱을 디바운스.

### 제안
- 스트리밍 커서 `▌` 추가 (CSS blink animation)
- 스트리밍 중에는 plain text, 완료 후 마크다운 렌더링으로 전환
- 스트리밍 중 자동 스크롤 보장 (onToken 콜백에서 scrollToBottom 호출)

## 3. 에스컬레이션 플로우 UX (High)

**정민수**: 에스컬레이션은 이 챗봇의 핵심 차별점인데, 현재 플로우가 너무 길고 단계가 많다:

```
담당자 연결하기 클릭
→ "담당자를 연결하고 있습니다..." (시스템)
→ 채소희 입장 + 인사 (1.5초 대기)
→ 각 담당자 입장 시스템 메시지 (1초 × N명)
→ Slack 채널 메시지 표시
→ Slack 질문 전송
→ 담당자 답변 폴링 (최대 90초)
→ "담당자 답변이 도착했습니다 ✅"
→ Continue/End 버튼 표시
```

이게 최소 10초, 최대 90초+ 걸린다. 그 동안 사용자는 입력도 못 하고(`isEscalationBusy`) 기다려야 한다.

**김도현**: B2B 상담에서 "담당자 연결"은 전환의 핵심 순간이다. 여기서 이탈하면 리드를 잃는다. 현재 UX는:
1. 대기 시간이 너무 길다
2. 진행 상황 피드백이 부족하다 (시스템 메시지만 있고 프로그레스 바 없음)
3. 대기 중 사용자가 할 수 있는 게 없다

### 제안
- 에스컬레이션 시작 즉시 입력 활성화 — 사용자가 추가 컨텍스트를 미리 입력할 수 있게
- 담당자 입장 시스템 메시지를 AgentStatus 패널로 통합 — 채팅 영역을 시스템 메시지로 오염시키지 않기
- 프로그레스 인디케이터 추가 — "담당자 연결 중 (2/3명 입장)" 같은 실시간 상태

## 4. 메시지 버블 디자인 일관성 (Medium)

**최유진**: 메시지 타입별 스타일이 너무 다양해서 시각적 일관성이 떨어진다:

| 타입 | 배경색 | 테두리 | 모서리 |
|------|--------|--------|--------|
| user | blue-600→indigo-600 gradient | 없음 | rounded-lg |
| ai | white→blue-50 gradient | blue-100 | rounded-lg |
| agent | blue-50 | blue-200 | rounded-lg |
| system | gray-100 | 없음 | rounded-lg |
| slack | purple-50 | purple-400 (left-4) | rounded-r-lg |
| contactCard | white | gray-200 | rounded-lg |

6가지 다른 스타일이 하나의 채팅 창에 섞인다. 사용자는 "이 메시지가 AI인지, 담당자인지, 시스템인지" 매번 시각적으로 판단해야 한다.

**박서연**: 특히 `ai`와 `agent` 메시지가 비슷한 위치(왼쪽)에 비슷한 색상(blue 계열)으로 나온다. 에스컬레이션 전환 시 "지금 AI가 답하는 건지, 사람이 답하는 건지" 혼동이 생긴다.

### 제안
- AI 메시지: 아이콘(🤖) + "AI" 라벨 명시
- Agent 메시지: 이미 아바타+이름이 있어서 OK, 하지만 `LIVE` 배지를 더 눈에 띄게
- System 메시지: 현재 italic + gray로 충분히 구분됨 ✅
- Slack 메시지: 보라색 계열이 갑자기 등장하면 이질적 — 에스컬레이션 컨텍스트에서만 보이므로 blue 계열로 통일하되 Slack 아이콘으로 구분

## 5. 모바일 UX (High)

**최유진**: 모바일에서 심각한 문제들이 있다:

### 5-1. 전체 화면 점유
```css
w-full h-[100dvh] /* 모바일에서 전체 화면 */
sm:w-[420px] sm:h-[700px] /* 데스크톱에서 고정 크기 */
```
모바일에서 챗봇이 전체 화면을 차지하는 건 좋은데, 뒤로가기 제스처가 없다. 닫기 버튼만 있고, 모바일 사용자가 기대하는 스와이프 다운 닫기가 없다.

### 5-2. 입력 영역 키보드 겹침
모바일에서 키보드가 올라오면 `h-[100dvh]`가 키보드를 포함한 높이인지 아닌지 브라우저마다 다르다. iOS Safari에서 입력 필드가 키보드에 가려질 수 있다.

### 5-3. Quick Reply 칩 터치 영역
```css
text-xs px-4 py-2 /* 터치 타겟이 작다 */
```
모바일 터치 타겟 최소 권장 크기는 44×44px인데, 현재 Quick Reply 버튼은 그보다 작다.

### 5-4. IntakeForm 입력 필드
모바일에서 4개 필드(기업명, 담당자명, 이메일, 연락처)를 스크롤 없이 보기 어렵다. 특히 키보드가 올라오면 폼 하단이 잘린다.

### 제안
- Quick Reply 칩: `py-2` → `py-3` (모바일에서 터치 영역 확보)
- IntakeForm: 모바일에서 2단계로 분리하거나, 필수 필드(기업명+이메일)만 먼저 받기
- 키보드 대응: `visualViewport` API 활용하여 키보드 높이만큼 패딩 조정

## 6. CollapsibleAIMessage 접기/펼치기 (Medium)

**박서연**: 6줄 이상 또는 300자 이상이면 자동 접기하는데, 이 기준이 문제다:

1. **마크다운 렌더링 전 줄 수 계산**: `text.split('\n')`으로 줄 수를 세는데, 마크다운 리스트(`- 항목`)는 렌더링 후 더 많은 시각적 줄을 차지한다. 4줄짜리 마크다운 리스트가 렌더링 후 8줄이 될 수 있다.

2. **접힌 상태에서 `...` 표시**: `lines.slice(0, 4).join('\n') + '...'`인데, 마크다운이 중간에 잘리면 깨진 마크다운이 렌더링된다. 예: `**볼드 텍스트가 여기서 잘리면...` → 닫히지 않은 bold.

3. **스트리밍 중 접기**: 스트리밍으로 텍스트가 늘어나는 중에 300자를 넘으면 갑자기 접힌다. 사용자가 읽고 있는 중에 텍스트가 사라지는 경험.

### 제안
- 스트리밍 중에는 접기 비활성화 (`isStreaming` 체크)
- 마크다운 잘림 방지: 문단 단위로 자르기 (빈 줄 `\n\n` 기준)
- 접기 기준을 렌더링 후 높이(px)로 변경 — `useRef`로 실제 높이 측정

## 7. 에러 처리 UX (Medium)

**정민수**: 에러 시 현재 동작:
```
"죄송합니다, 현재 AI 응답 생성에 일시적인 문제가 발생했습니다.
담당자를 직접 연결해 드릴 수 있습니다."
+ [🔄 다시 시도] 버튼
+ [전문 담당자 연결하기] 버튼
```

이건 괜찮은데, 문제는:
1. **스트리밍 실패 시 부분 텍스트 처리**: `prev.filter(m => !m.isStreaming)`으로 스트리밍 중이던 메시지를 제거하는데, 사용자가 이미 읽고 있던 부분 답변이 갑자기 사라진다.
2. **네트워크 에러 vs API 에러 구분 없음**: 인터넷 끊김과 서버 에러가 같은 메시지로 표시된다.
3. **재시도 횟수 제한 없음**: 사용자가 무한히 재시도할 수 있다.

### 제안
- 스트리밍 실패 시 부분 텍스트 유지 + "답변 생성이 중단되었습니다" 표시
- 네트워크 에러: "인터넷 연결을 확인해 주세요" / API 에러: "잠시 후 다시 시도해 주세요"
- 재시도 3회 후 자동으로 에스컬레이션 제안

## 8. 세션 관리 UX (Medium)

**김도현**: 세션 관련 몇 가지:

### 8-1. 닫기 버튼 = 새 대화
```jsx
onClick={() => { setIsOpen(false); handleNewChat() }}
```
X 버튼을 누르면 챗봇이 닫히면서 모든 대화가 초기화된다. 사용자가 실수로 X를 누르면 진행 중이던 상담이 전부 날아간다. 특히 에스컬레이션 중이었다면 Slack 채널은 열려있는데 프론트엔드 상태만 초기화되는 불일치가 발생한다.

**정민수**: 이건 심각하다. "닫기"와 "새 대화"는 완전히 다른 의도인데 하나의 버튼에 묶여있다.

### 8-2. 최소화 후 복원
최소화(`setIsMinimized(true)`)는 상태를 유지하지만, 닫기는 초기화한다. 사용자가 이 차이를 인지하기 어렵다.

### 제안
- X 버튼: 최소화와 동일하게 동작 (상태 유지, 챗봇만 숨김)
- "새 대화" 버튼: 헤더의 `+` 버튼으로 분리 (현재도 있지만, X와 역할이 겹침)
- 에스컬레이션 중 닫기 시 확인 다이얼로그: "상담이 진행 중입니다. 종료하시겠습니까?"

## 9. 입력 필드 상태 관리 (Low)

**이준호**: placeholder 텍스트가 상태에 따라 바뀌는데:
```
showIntakeForm → '위 정보를 입력해 주세요'
isAIProcessing → 'AI 답변 생성 중...'
isEscalationBusy → '담당자 연결 중...'
sessionClosed → '상담이 종료되었습니다'
showContinueOrEnd → '위 버튼을 선택해 주세요'
escalationMode → '담당자에게 질문하기...'
default → '메시지를 입력하세요...'
```

7가지 상태인데, 우선순위가 삼항 연산자 체인으로 되어있어서 조건이 겹칠 때 예상치 못한 placeholder가 나올 수 있다. 예: `isAIProcessing`이면서 `escalationMode`이면 'AI 답변 생성 중...'이 나오는데, 에스컬레이션 모드에서 AI 처리가 동시에 일어나는 경우는 없으므로 실제 문제는 아니지만, 코드 가독성이 떨어진다.

### 제안
- placeholder 로직을 별도 함수로 분리
- 상태 우선순위를 명시적으로 문서화

## 10. 접근성(Accessibility) (Medium)

**이준호**: 접근성 관점에서:

1. **aria-label 부족**: 메시지 버블에 role="log"나 aria-live="polite"가 없다. 스크린 리더가 새 메시지를 인식하지 못한다.
2. **키보드 네비게이션**: Quick Reply 칩, 에스컬레이션 버튼 등이 Tab 키로 접근 가능하지만, focus 스타일이 `focus:ring-2`만 있고 고대비 모드에서 보이지 않을 수 있다.
3. **색상 대비**: system 메시지의 `text-gray-500`은 `bg-gray-100` 위에서 WCAG AA 기준(4.5:1)을 충족하지 못할 수 있다.
4. **스트리밍 텍스트**: 스크린 리더가 매 토큰마다 읽으면 사용자 경험이 매우 나쁘다. `aria-live="off"`로 스트리밍 중에는 비활성화하고, 완료 후 `aria-live="polite"`로 전체 답변을 한 번에 읽어야 한다.

### 제안
- 메시지 영역에 `role="log"` + `aria-live="polite"` 추가
- 스트리밍 중 `aria-busy="true"` 설정
- system 메시지 색상 대비 개선: `text-gray-600` 이상

## 11. ThinkingPanel과 답변의 시간 불일치 (High)

**박서연**: 코드를 보면:
```js
if (thinkingStartTime) {
  const elapsed = Date.now() - thinkingStartTime
  if (elapsed < 1500) await new Promise(r => setTimeout(r, 1500 - elapsed))
}
setShowThinking(false)
```

스트리밍이 시작되면 토큰이 바로 오는데, ThinkingPanel은 최소 1.5초 동안 유지된다. 즉:
- 0ms: 질문 전송
- 200ms: meta 이벤트 도착 → ThinkingPanel 업데이트
- 500ms: 첫 토큰 도착 → 답변 텍스트 시작
- 1500ms: ThinkingPanel 닫힘

**500ms~1500ms 사이에 답변이 스트리밍되면서 동시에 ThinkingPanel도 보인다.** 이게 스크린샷에서 보이는 문제의 원인이다.

### 제안
- 첫 토큰 도착 시 즉시 ThinkingPanel 닫기 (1.5초 최소 대기 제거)
- 또는 ThinkingPanel을 답변 버블 내부로 통합하여 자연스럽게 전환

---

## 12. Quick Reply UX (Low)

**최유진**: Quick Reply 칩이 수평 스크롤인데:
1. 스크롤 가능하다는 시각적 힌트가 약하다 (오른쪽 그라데이션만 있음)
2. `followUpIndex`로 5개씩 보여주는데, 다음 5개를 볼 방법이 없다
3. 에스컬레이션 서브질문 기반 Quick Reply는 좋은 아이디어지만, 질문이 길면 칩이 너무 넓어진다

### 제안
- 긴 질문은 말줄임(...) 처리 + 툴팁
- 수평 스크롤 대신 2줄 그리드 레이아웃 (모바일에서 더 자연스러움)
